---
title: Paging options
---

Paging options transform a paged field from a prototype into a reliable API contract. They let you set default page sizes, limit expensive requests, require clients to specify page sizes, control when counts are exposed, select paging providers, and keep generated type names consistent.

Begin with schema-wide defaults, then override individual cursor or offset fields as needed for cost, provider, or naming differences.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options =>
    {
        options.DefaultPageSize = 25;
        options.MaxPageSize = 100;
        options.RequirePagingBoundaries = true;
        options.IncludeTotalCount = false;
    });
```

These settings apply to both cursor and offset paging where relevant. For initial setup, see [Pagination](index.md), [Cursor connections](connections-cursor.md), or [Offset paging](offset.md).

# Configuration levels and precedence

Hot Chocolate applies paging options from the most specific to the least specific configuration.

```text
field option
    overrides
ModifyPagingOptions(...)
    overrides
PagingDefaults
```

The built-in defaults are:

| Option                                | Effective default |
| ------------------------------------- | ----------------- |
| `DefaultPageSize`                     | `10`              |
| `MaxPageSize`                         | `50`              |
| `IncludeTotalCount`                   | `false`           |
| `AllowBackwardPagination`             | `true`            |
| `RequirePagingBoundaries`             | `false`           |
| `InferConnectionNameFromField`        | `true`            |
| `InferCollectionSegmentNameFromField` | `true`            |
| `ProviderName`                        | `null`            |
| `IncludeNodesField`                   | `true`            |
| `EnableRelativeCursors`               | `false`           |
| `NullOrdering`                        | `Unspecified`     |

Unset field-level values inherit the global value. If `DefaultPageSize` exceeds `MaxPageSize`, Hot Chocolate uses `MaxPageSize` as the effective default for that field.

# Configure global paging defaults

Apply `.ModifyPagingOptions(...)` on the GraphQL builder when you want many fields to share the same production rules.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options =>
    {
        options.DefaultPageSize = 25;
        options.MaxPageSize = 100;
        options.RequirePagingBoundaries = true;
        options.IncludeTotalCount = false;
    });
```

This configuration results in:

- A cursor field without `first` or `last` returns 25 items, unless boundaries are required.
- An offset field without `take` returns 25 items, unless boundaries are required.
- Clients cannot request more than 100 items with `first`, `last`, or `take`.
- Clients must specify a page size.
- `totalCount` is excluded from generated result types unless a field opts in.

`MaxPageSize` also impacts Hot Chocolate [cost analysis](/docs/hotchocolate/v16/build/security/cost-analysis), as cost calculation uses it as the assumed list size for paged fields.

# Override a single cursor field

Use `[UsePaging]` when the configuration is brief and belongs with the resolver.

```csharp
using HotChocolate.Types;

[QueryType]
public static partial class ProductQueries
{
    [UsePaging(DefaultPageSize = 25, MaxPageSize = 100, IncludeTotalCount = true)]
    public static IQueryable<Product> GetProducts(CatalogContext db)
    {
        return db.Products.OrderBy(p => p.Name).ThenBy(p => p.Id);
    }
}
```

Use a descriptor when schema configuration is managed outside the resolver, generated by code, or requires `PagingOptions` members not available on the attribute.

```csharp
using HotChocolate.Types;
using HotChocolate.Types.Pagination;

public sealed class QueryType : ObjectType
{
    protected override void Configure(IObjectTypeDescriptor descriptor)
    {
        descriptor
            .Field("products")
            .UsePaging(options: new PagingOptions
            {
                DefaultPageSize = 25,
                MaxPageSize = 100,
                IncludeTotalCount = true
            });
    }
}
```

`ConnectionName` is a convenience argument and attribute property. It is not a `PagingOptions` property.

# Override one offset field

Offset paging uses `skip` and `take`, and returns a collection segment.

```csharp
using HotChocolate.Types;

[QueryType]
public static partial class ProductQueries
{
    [UseOffsetPaging(DefaultPageSize = 25, MaxPageSize = 100, IncludeTotalCount = true)]
    public static IQueryable<Product> GetProducts(CatalogContext db)
    {
        return db.Products.OrderBy(p => p.Name).ThenBy(p => p.Id);
    }
}
```

The fluent form uses `.UseOffsetPaging(...)`.

```csharp
using HotChocolate.Types;
using HotChocolate.Types.Pagination;

public sealed class QueryType : ObjectType
{
    protected override void Configure(IObjectTypeDescriptor descriptor)
    {
        descriptor
            .Field("products")
            .UseOffsetPaging(options: new PagingOptions
            {
                DefaultPageSize = 25,
                MaxPageSize = 100,
                IncludeTotalCount = true
            });
    }
}
```

`CollectionSegmentName` is a convenience argument and attribute property. It is not a `PagingOptions` property.

# Page size recipes

## Set the default page size

Set a global default when most paged fields should return the same number of items without an explicit client boundary.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options => options.DefaultPageSize = 25);
```

Override a field when its cost or UX differs.

```csharp
[UsePaging(DefaultPageSize = 10)]
public static IQueryable<Product> GetFeaturedProducts(CatalogContext db)
{
    return db.Products.Where(p => p.IsFeatured).OrderBy(p => p.Id);
}
```

Behavior:

| Field kind | Client omits boundary | Result                                                              |
| ---------- | --------------------- | ------------------------------------------------------------------- |
| Cursor     | No `first` or `last`  | Uses `DefaultPageSize`, unless `RequirePagingBoundaries` is `true`. |
| Offset     | No `take`             | Uses `DefaultPageSize`, unless `RequirePagingBoundaries` is `true`. |

## Limit maximum page size

Use a conservative global maximum and lower limits for expensive fields.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options => options.MaxPageSize = 100);
```

```csharp
[UsePaging(MaxPageSize = 20)]
public static IQueryable<Review> GetRecentReviews(CatalogContext db)
{
    return db.Reviews.OrderByDescending(r => r.CreatedAt).ThenBy(r => r.Id);
}
```

If a client sends `first`, `last`, or `take` above the maximum, Hot Chocolate returns a GraphQL error. Raise the maximum only after considering resolver cost and cost-analysis budgets.

## Require clients to request a page size

Use `RequirePagingBoundaries` when public clients should state how many items they want.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options => options.RequirePagingBoundaries = true);
```

Valid cursor request:

```graphql
query GetProducts {
  products(first: 25) {
    nodes {
      id
    }
  }
}
```

Invalid cursor request when boundaries are required:

```graphql
query GetProducts {
  products {
    nodes {
      id
    }
  }
}
```

Valid offset request:

```graphql
query GetProducts {
  products(take: 25) {
    items {
      id
    }
  }
}
```

Invalid offset request when boundaries are required:

```graphql
query GetProducts {
  products {
    items {
      id
    }
  }
}
```

# Total count recipes

`totalCount` is not generated by default. Enable it on fields where clients have a product requirement for counts.

```csharp
[UsePaging(IncludeTotalCount = true)]
public static IQueryable<Product> GetProducts(CatalogContext db)
{
    return db.Products.OrderBy(p => p.Id);
}
```

```csharp
[UseOffsetPaging(IncludeTotalCount = true)]
public static IQueryable<Product> GetProducts(CatalogContext db)
{
    return db.Products.OrderBy(p => p.Id);
}
```

Without `IncludeTotalCount`, the cursor connection does not include a count field.

```graphql
# Without IncludeTotalCount
type ProductsConnection {
  pageInfo: PageInfo!
  edges: [ProductsEdge!]
  nodes: [Product!]
}
```

With `IncludeTotalCount`, the generated type exposes the field.

```graphql
# With IncludeTotalCount
type ProductsConnection {
  pageInfo: PageInfo!
  edges: [ProductsEdge!]
  nodes: [Product!]
  totalCount: Int!
}
```

Offset collection segments follow the same option.

```graphql
# With IncludeTotalCount
type ProductsCollectionSegment {
  items: [Product!]
  pageInfo: CollectionSegmentInfo!
  totalCount: Int!
}
```

Clients must request `totalCount` for the server to compute or return it. Count cost depends on filters, joins, indexes, provider support, and resolver shape. When your resolver returns a manual `Connection<T>` or collection segment type, provide count behavior in that result shape.

# Naming recipes

## Override a cursor connection name

Use an explicit connection name when the same node type appears in multiple fields with different semantics, or when you plan to extend the generated type.

```csharp
[UsePaging(ConnectionName = "CatalogProducts")]
public static IQueryable<Product> GetProducts(CatalogContext db)
{
    return db.Products.OrderBy(p => p.Id);
}
```

```csharp
descriptor
    .Field("products")
    .UsePaging(connectionName: "CatalogProducts");
```

Hot Chocolate appends `Connection` and `Edge`, so this produces `CatalogProductsConnection` and `CatalogProductsEdge`.

## Change cursor naming inference globally

By default, the field name is used as the base for cursor connection and edge types. Disable that behavior only when you need compatibility with a schema that used older node-type-based names.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options => options.InferConnectionNameFromField = false);
```

## Override an offset collection segment name

```csharp
[UseOffsetPaging(CollectionSegmentName = "CatalogProducts")]
public static IQueryable<Product> GetProducts(CatalogContext db)
{
    return db.Products.OrderBy(p => p.Id);
}
```

```csharp
descriptor
    .Field("products")
    .UseOffsetPaging(collectionSegmentName: "CatalogProducts");
```

This produces `CatalogProductsCollectionSegment`.

## Change offset naming inference globally

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options => options.InferCollectionSegmentNameFromField = false);
```

# Cursor behavior options

## Disable backward pagination

Backward cursor pagination uses `last` and `before`. It is enabled by default.

```csharp
[UsePaging(AllowBackwardPagination = false)]
public static IQueryable<Product> GetProducts(CatalogContext db)
{
    return db.Products.OrderBy(p => p.Id);
}
```

You can also set it globally.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options => options.AllowBackwardPagination = false);
```

When backward pagination is disabled, clients use `first` and `after`. If `RequirePagingBoundaries` is also `true`, clients must provide `first`.

## Hide the nodes shortcut field

The cursor connection `nodes` field is a Hot Chocolate convenience shortcut. Disable it when you want clients to use `edges { node cursor }` consistently.

```csharp
builder
    .AddGraphQL()
    .ModifyPagingOptions(options => options.IncludeNodesField = false);
```

Or configure a descriptor field.

```csharp
descriptor
    .Field("products")
    .UsePaging(options: new PagingOptions
    {
        IncludeNodesField = false
    });
```

Before:

```graphql
type ProductsConnection {
  pageInfo: PageInfo!
  edges: [ProductsEdge!]
  nodes: [Product!]
}
```

After:

```graphql
type ProductsConnection {
  pageInfo: PageInfo!
  edges: [ProductsEdge!]
}
```

The `[UsePaging]` attribute does not expose `IncludeNodesField` directly.

## Configure null ordering for cursor keys

When cursor ordering includes nullable keys, the paging provider needs to know how the data source orders null values.

```csharp
using HotChocolate.Types.Pagination;

builder
    .AddGraphQL()
    .ModifyPagingOptions(options =>
    {
        options.NullOrdering = NullOrdering.NativeNullsLast;
    });
```

Use `NativeNullsFirst` or `NativeNullsLast` to match the database. Leave `Unspecified` when provider detection covers your data source. See [Cursor connections](connections-cursor.md) for cursor-key guidance.

## Relative cursor and page info fields

`EnableRelativeCursors`, `RelativeCursorFields`, and `PageInfoFields` are advanced cursor options. They are available through global or fluent `PagingOptions`; `EnableRelativeCursors` is also exposed on `[UseConnection]`. They are not standard `[UsePaging]` attribute properties.

Use these options only when your connection implementation needs relative cursor behavior or custom field-selection detection. See [Cursor connections](connections-cursor.md) for connection-specific patterns.

# Provider selection

`ProviderName` selects a named paging provider for a field. If it is omitted, Hot Chocolate infers the provider from the source type where possible. If no provider can be inferred, registered provider defaults and order decide the fallback.

```csharp
builder
    .AddGraphQL()
    .AddMongoDbPagingProviders(providerName: "MongoDB");
```

```csharp
using HotChocolate;
using HotChocolate.Data;
using HotChocolate.Types;
using MongoDB.Driver;

[UsePaging(ProviderName = "MongoDB")]
public static IExecutable<Product> GetProducts(IMongoCollection<Product> products)
{
    return products.AsExecutable();
}
```

Keep provider registration and custom provider implementation on the provider page. Continue with [Paging providers](paging-providers.md) for MongoDB, Raven, queryable providers, and custom providers.

# Middleware order gotcha

When paging is combined with projections, filtering, and sorting, use this order:

```csharp
[UsePaging]
[UseProjection]
[UseFiltering]
[UseSorting]
public static IQueryable<Product> GetProducts(CatalogContext db)
{
    return db.Products;
}
```

Incorrect order can change which data is paged, filtered, sorted, or projected. The same rule applies when configuring middleware through descriptors.

# Choose attributes or descriptors

Use attributes when the option set is local, static, and short.

```csharp
[UsePaging(DefaultPageSize = 25, MaxPageSize = 100)]
public static IQueryable<Product> GetProducts(CatalogContext db)
{
    return db.Products.OrderBy(p => p.Id);
}
```

Use descriptors when:

- Configuration belongs outside the resolver class.
- Multiple fields share generated configuration.
- You need `PagingOptions` members that an attribute does not expose.
- The field is configured on a third-party CLR type.
- You want one schema type to show all middleware in one place.

```csharp
descriptor
    .Field("products")
    .UsePaging(options: new PagingOptions
    {
        IncludeNodesField = false,
        NullOrdering = NullOrdering.NativeNullsLast
    });
```

# Safe production defaults

Use these as starting points, then tune per field:

- Start with `DefaultPageSize` between 10 and 25 for public fields.
- Keep `MaxPageSize` conservative, often 50 or 100 depending on resolver cost.
- Use lower `MaxPageSize` values on nested or expensive fields.
- Set `RequirePagingBoundaries = true` when clients should be explicit and cost should be predictable.
- Keep `IncludeTotalCount = false` globally. Enable it only where clients need counts.
- Use deterministic ordering with a unique final key, such as `Id`, before applying paging.

# Troubleshooting

| Symptom                                               | Likely cause                                                                          | Fix                                                                                                                           |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| Query fails when requesting too many items.           | `first`, `last`, or `take` exceeds `MaxPageSize`.                                     | Request a smaller page, raise the max cautiously, or configure a higher field-level max only for that field.                  |
| Query fails when omitting `first`, `last`, or `take`. | `RequirePagingBoundaries = true`.                                                     | Provide the boundary, or disable the requirement only for trusted fields.                                                     |
| `last` or `before` is not accepted.                   | `AllowBackwardPagination = false`.                                                    | Use `first` and `after`, or enable backward pagination for that cursor field.                                                 |
| Connection or collection segment type names changed.  | Field-name inference is enabled.                                                      | Set `ConnectionName` or `CollectionSegmentName`, or set the matching inference option to `false` for migration compatibility. |
| `totalCount` is missing.                              | `IncludeTotalCount` defaults to `false`.                                              | Enable it globally or on the field, and provide count behavior for manual result types.                                       |
| `nodes` is missing.                                   | `IncludeNodesField = false`.                                                          | Query through `edges { node }` or re-enable `IncludeNodesField`.                                                              |
| The wrong paging provider is used.                    | Missing `ProviderName`, mismatched provider registration, or unsupported source type. | Set the field `ProviderName`, register the needed provider, or return a supported source shape.                               |
| Attribute does not have the option you expected.      | Not every `PagingOptions` member is exposed on every attribute.                       | Use fluent `PagingOptions` or global `.ModifyPagingOptions(...)`, then check the option table.                                |
