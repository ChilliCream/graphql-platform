Mocha topology: what was done and why

You asked why MochaSandbox emitted convention topology despite fully explicit configuration. I diagnosed the cause, fixed the framework and the sandbox, and verified it through adversarial review. Everything is uncommitted.

Diagnosis: explicit mode only half existed, so Mocha generated topology you never declared
BindHandlersExplicitly() never reached the topology conventions
  • ConsumerBindingMode.Explicit was read in exactly one place (MessagingTransport.Lifecyle.cs), which only stops endpoint auto-creation.
  • RabbitMQReceiveEndpointTopologyConvention ran for every endpoint and every inbound route regardless.
  • For each consumed type it synthesized the chain: publish exchange → send exchange → queue (e.g. contracts.order-placed → order-placed → orderservice.events.queue; same for order-cancelled, start-order-saga, get-stock-info-request/result).
  • Nothing checked whether explicit bindings already covered the message type.
AutoProvision(false) gated broker declaration only, not the model or the snapshot
  • Convention entities are created with AutoProvision = null and resolve entity ?? transportDefault only inside ProvisionTopologyAsync.
  • Describe() exported the entire model unfiltered, so the phantoms appeared in your JSON with autoProvision: false.
  • Your flags worked as implemented; they just operated one layer below where you expected.
Outbound routes were fabricated for every consumed type, defaulting to convention exchanges
  • CreateMatchingOutboundRoute maps subscribe → publish and send/request → send for every inbound route.
  • OrderService had no AddMessage<T>().Publish/Send destinations (those live only in the Api project), so its routes defaulted to e/contracts.order-placed etc.
  • Runtime trap: publishing one of those would target an exchange that AutoProvision(false) guaranteed never exists.
Two oddities were bugs of their own
  • The doubled response.stock.get-info bind was declared twice in the sandbox's Program.cs (saga section and RPC section); Mocha did not dedupe.
  • The bind URN ignored the routing key, so the three events.exchange → events.queue binds shared one resource id and id-keyed registries would drop two.
Framework fixes: explicit mode now means the model contains only what you declared (hc3/src/Mocha, uncommitted, 36 files +400/−71 plus 5 new)
1. Topology conventions skip convention chains in explicit mode
  • Both RabbitMQ conventions (receive and dispatch) early-return when the mode is Explicit.
  • Mode exposed via a MessagingTransport.ConsumerBindingMode property assigned in Initialize, because Configuration is nulled in Finalize and conventions run later.
  • Reply queue creation happens before the early-return, so request/reply keeps working.
  • Implicit mode is snapshot-pinned as unchanged.
2. No eagerly fabricated outbound routes in explicit mode; producers behave the same in both modes
  • Both CreateMatchingOutboundRoute call sites are gated on Implicit, so the snapshot no longer contains outbound routes and dispatch endpoints nobody configured.
  • Dispatching a message with no configured route lazily derives the convention destination in both modes; with AutoProvision(false) the mandatory: true publish fails loudly at the broker if the target exchange is absent.
  • Design reversal: an earlier iteration made unconfigured dispatch throw in explicit mode, scoped via an ExplicitRouteEnforcementFeature marker. The owner rejected it: ConsumerBindingMode is a consumer-side knob and must not change runtime producer behavior. The throw, the feature, and its ThrowHelper entry were removed; the router is byte-identical to HEAD again.
  • Replies were never affected either way: they resolve by address through transport.ReplyDispatchEndpoint and bypass route resolution entirely, now pinned by a runtime test.
3. Routing key and arguments became part of bind identity; duplicate declarations dedupe
  • Bind addresses now carry ?rk={escaped-routing-key}, fixing the URN collision.
  • Headers bindings (distinguished only by arguments) get an ?args={hash} discriminator from a collision-resistant canonical form: escaped separators, type tags, null distinct from empty string, byte[] by content.
  • Dedupe uses structural, type-sensitive argument equality ({x:1} ≠ {x:"1"}), so distinct bindings are never silently dropped.
  • Merging true duplicates: non-null AutoProvision beats null, true wins conflicts, declared provenance beats convention.
4. Error/skip queues inherit the declared parent queue's AutoProvision
  • Previously they resolved to the transport default, so with AutoProvision(false) faulted messages would dead-letter into queues nobody creates.
  • Now orders_error/orders_skipped resolve exactly like the queue they serve; falls back to the transport default when the queue is not declared.
5. Every Describe() entity and link carries provenance: source: declared | convention
  • Tracked on the model objects via shared constants, so the Nitro visualizer can distinguish your topology from generated topology.
  • Chosen over filtering, since autoProvision: false legitimately means "exists, managed externally".
6. API honesty: removed the dead saga AutoProvision flag
  • SagaTransitionDescriptor.AutoProvision removed entirely: it never affected topology, and wiring it would thread four layers for a dead prerelease flag.
  • A RemoveConvention<T>() escape hatch was built and then rolled back: mode gating already solves the reported problem, and its headline use (dropping the receive topology convention) breaks endpoint completion because that convention is load-bearing.
  • A deliberate convention replace/configure API is deferred until the load-bearing receive convention is split, so removal is not a footgun.
Hardening picked up along the way
  • ConsumerBindingMode.Implicit is now enum value 0, so an uninitialized read fails toward the safe default instead of silently skipping conventions.
  • The internal ?autoProvision= query parse accepts the parameter at any position.
  • An XML doc that overclaimed (AutoProvision "queue or exchange") now states the real contract.
Sandbox fix: removed the self-inflicted duplicate binding (MochaSandbox, uncommitted)
  • Deleted the RPC-section copy of DeclareBinding(events.exchange, orderservice.saga.queue).RoutingKey(response.stock.get-info) in OrderService/Program.cs; the saga-section declaration remains the single one.
  • Build clean: 0 warnings, 0 errors. Your pre-existing uncommitted edits were left untouched.
  • Caveat: the sandbox consumes NuGet 16.1.4-p.4, so it will not reflect the framework fixes until packages are rebuilt from the patched source.
Verification: implementer plus devil's advocate over three rounds, 1,800+ tests including real brokers
Round 1 review found one MAJOR and a set of minors
  • MAJOR: the new binding dedupe ignored Arguments, silently collapsing headers-exchange bindings (a regression this diff introduced).
  • Minors: untested RemoveConvention, enum default hazard, descriptor API asymmetry, doc inaccuracies, brittle query parse.
  • All lifecycle and gating claims were independently verified, including against a real Docker RabbitMQ broker (303/303).
Round 2 fixed everything; re-review caught corner cases in the new equality logic
  • Canonicalization was still type-blind: {x:1} vs {x:"1"}, null vs empty, separator injection, and byte[] values colliding on their type name.
  • Verdict was already ship-ready, but silent binding loss is the exact bug class we set out to kill, so a final round closed it.
Round 3 closed the remaining items with structural equality and direct merge tests
  • Type-sensitive structural dedupe, collision-resistant canonical form for the URN hash, MergeFrom conflict rule (true wins) under direct tests, load-bearing-conventions doc warning.
Round 4 reversed the dispatch throw after the owner challenged its semantics
  • The reply path was traced end to end and proven address-based (ReplyAsync → transport.ReplyDispatchEndpoint), so the throw never threatened replies, but it overloaded a consumer-side mode with producer-side enforcement.
  • Throw, marker feature, and ThrowHelper entry removed; IMessageRouter.cs and ThrowHelper.cs are back to HEAD content.
  • New tests pin the final contract: lazy convention derivation for unconfigured types in explicit mode, and a runtime InMemory request/reply round trip under BindHandlersExplicitly().

  • Docker integration suites (RabbitMQ 308/308, Postgres 293/293) last ran before the reversal, which only restored two core files to HEAD content.
